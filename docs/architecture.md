## Patrones de diseГұo

| PatrГіn | Uso |
|--------|-----|
| **DTO** | SeparaciГіn entre entidades JPA y datos de la API |
| **Repository** | AbstracciГіn de persistencia con Spring Data JPA |
| **Specification** | Filtros dinГЎmicos sin SQL nativo |
| **Mapper** | MapStruct para conversiГіn Entity вҶ” DTO |
| **Builder** | ConstrucciГіn de objetos complejos (Lombok) |
| **Strategy** | Roles y permisos vГӯa Spring Security |
| **Interceptor** | JWT filter en la cadena de filtros HTTP |
| **Template Method** | BaseEntity con auditorГӯa automГЎtica |

## Decisiones tГ©cnicas

### ВҝPor quГ© UUID en lugar de Long?
- IDs Гәnicos globales sin depender de secuencias
- Previene enumeraciГіn de recursos
- Adecuado para sistemas distribuidos

### ВҝPor quГ© soft-delete?
- Los vehГӯculos no se eliminan fГӯsicamente
- Se marcan como `deleted = TRUE`
- Todas las consultas filtran `WHERE deleted = FALSE`

### ВҝPor quГ© `ddl-auto: validate`?
- Control total del esquema vГӯa `schema.sql`
- Hibernate valida que las entidades coincidan con la BD
- Evita migraciones automГЎticas en producciГіn

### ВҝPor quГ© MapStruct y no manual mapping?
- Tipado fuerte en compilaciГіn
- Cero overhead en runtime
- ReducciГіn de cГіdigo boilerplate
